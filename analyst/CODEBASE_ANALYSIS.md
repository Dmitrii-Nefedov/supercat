# Codebase Analysis: Supercat Weather

> Analyst: @supercat-agent | Date: 2026-07-05

## 1. Executive Summary

Supercat Weather is a single-page weather forecast PWA built with zero dependencies — pure HTML/CSS/JS. It consumes the free Open-Meteo API for weather data and geocoding. The entire frontend lives in **one file** (`index.html`, 1673 lines) containing inline CSS (~990 lines) and inline JS (~680 lines). A Python CLI companion (`weather.py`, 239 lines) mirrors the web app's core functionality.

**Maturity**: Production-ready static app. Feature-complete for v1. Blocked only on GitHub Pages deployment (PAT scope issue).

## 4. Component Map (index.html)

### 4.1 CSS Architecture (~990 lines)

- **Base styles** (lines 27-55): Reset, body, container, card — glassmorphism via `backdrop-filter`
- **Search** (lines 63-171): Input, locate button, dropdown with autocomplete
- **Header & weather display** (lines 173-255): City name, date, temperature, icon, condition
- **Detail grid** (lines 257-289): 8-item grid — humidity, wind, pressure, cloud cover, sunrise, sunset, UV index, wind direction
- **Forecast** (lines 291-403): 7-day daily forecast, hourly scroll, forecast toggle (daily/hourly)
- **Weather backgrounds** (lines 425-548): 7 themes — `sunny`, `clear-night`, `cloudy`, `foggy`, `rainy`, `snowy`, `thunderstorm`
- **Controls** (lines 550-830): Unit toggle (°C/°F), theme toggle (dark/light), light theme overrides
- **Animations** (lines 892-989): `spinSlow` (sun), `cloudDrift`, `rainBounce`, `flash` (thunder), `snowDrift`, `fogPulse`, `twinkle` (night)
- **Accessibility** (lines 939-950): `focus-visible` styles, `aria-selected` highlight
- **Reduced motion** (lines 883-890): `@media (prefers-reduced-motion: reduce)` disables all animations
- **Loading bar** (lines 991-1014): Thin animated bar at card top during refresh

### 4.2 JavaScript Architecture (~680 lines)

**State** (lines 1065-1123):
- `selectedCity` — stored in localStorage (`supercat_weather_city`)
- `currentUnit` — `'C'` or `'F'` (`supercat_weather_unit`)
- `currentTheme` — `'dark'` or `'light'` (`supercat_weather_theme`)

**Data flow**:
```
fetchWeather(city) → Open-Meteo API → JSON → renderWeather(data, city) → DOM
searchCities(query) → Geocoding API → city list → dropdown → selectCity → fetchWeather
```

## 5. PWA Infrastructure

| File | Role |
|------|------|
| `sw.js` | Cache v3: installs index.html, manifest.json, 404.html; network-first navigation; cache-first static assets; falls back to 404.html |
| `manifest.json` | `display: standalone`, purple theme, SVG inline icons |
| `index.html <meta>` | `viewport-fit=cover`, `apple-mobile-web-app-capable=yes`, theme-color |

**SW strategy**: Navigate requests use network-first (fallback to cache). Same-origin static requests use cache-first with background refresh. Cross-origin (API) requests are not cached.

## 7. Dependency Analysis

### External Dependencies: NONE
- Zero npm packages, no build tools, no frameworks
- Google Fonts (`Inter`) loaded via `<link>` (not `@import`)
- Open-Meteo API — no API key required, rate-limited only

### Internal Dependencies
- `index.html` → `manifest.json` (link rel=manifest)
- `index.html` → `sw.js` (serviceWorker.register)
- `sw.js` → `index.html`, `manifest.json`, `404.html` (cache.addAll)
- `index.html` → Open-Meteo Forecast API (runtime HTTP)
- `index.html` → Open-Meteo Geocoding API (runtime HTTP)
- `weather.py` → Open-Meteo APIs (runtime HTTP)

### Dependency Graph
```
index.html ──→ manifest.json (manifest)
index.html ──→ sw.js (service worker)
sw.js ──→ index.html, manifest.json, 404.html (cache)
index.html ──→ api.open-meteo.com (weather + geocoding)
weather.py ──→ api.open-meteo.com (weather + geocoding)
```

No circular dependencies. Clean separation.

## 8. Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| **Monolithic index.html** | Zero build step, instant deploy to any static host |
| **Inline CSS + JS** | Single HTTP request for the app shell |
| **Glassmorphism** | `backdrop-filter: blur(20px)` — modern, visually rich, works cross-browser |
| **CSS-only weather overlays** | Rain/snow/lightning via `::before`/`::after` pseudo-elements — no Canvas overhead |
| **Russian UI** | Target audience. i18n would require string externalization |
| **localStorage persistence** | City, unit preference, theme preference — survives page reloads |
| **Service worker cache-first for same-origin** | Offline-capable PWA |
| **Network-first for navigation** | Always shows fresh data; falls back to cached shell |

## 9. Risk Assessment

### 🔴 High Risk
- **Single file bottleneck**: `index.html` at 1673 lines mixes HTML, CSS, and JS. This is fragile — any malformed change breaks the entire app. Splitting into separate files (or at minimum extracting CSS) is recommended for maintainability.
- **No automated testing**: Zero tests for either web app or CLI. Each push is a manual test.

### 🟡 Medium Risk
- **No error boundary for API failures**: `fetchWeather()` has a try/catch, but partial API failures (e.g., hourly data missing) could silently render incomplete UI.
- **localStorage quota**: On some mobile browsers, localStorage can be cleared under storage pressure. No graceful handling.
- **Geolocation timeout**: 10s timeout. On slow connections, the locate button stays in "loading" state.

### 🟢 Low Risk
- **No CSP headers**: Currently served without Content-Security-Policy (acceptable for static Pages hosting).
- **Emoji weather icons**: Emoji rendering varies by OS/browser. Custom SVG icons would be more consistent.
- **Hardcoded `Europe/Moscow` fallback**: If geolocation reverse-lookup fails, timezone defaults to Moscow.

## 10. Opportunities & Next Steps

### Immediate (no code changes needed)
1. Enable GitHub Pages via Settings → Pages → `Deploy from a branch` → `main` → `/` (root)

### Short-term (high impact, low effort)
2. **Extract CSS to separate file** — reduces index.html by ~60%, enables caching of styles separately
3. **Add basic smoking tests** — validate that HTML parses and JSON responses match expected schema
4. **Fix `weather.py` gaps** — add hourly forecast support, `uv_index_max` parameter, and unit toggle
5. **Add `.editorconfig`** — consistent indentation across contributors

### Medium-term
6. **WebSocket or Server-Sent Events** — real-time weather alerts from Open-Meteo
7. **i18n framework** — extract Russian strings to a JSON file, add English locale
8. **Custom SVG weather icons** — replace emoji with styled SVG sprites for cross-platform consistency
9. **CSS custom properties** — refactor repeated color values into `:root` variables for theming

### Long-term
10. **Svelte or Lit migration** — component-based architecture without a build step (single-file components with native `customElements`)
11. **End-to-end tests** — Playwright or Puppeteer for weather search flow, geolocation mock, and PWA install prompt
